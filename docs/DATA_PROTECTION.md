# Защита данных пользователей — ReviMarket

Документ описывает, какие данные есть в проекте **ReviMarket**, как их защищать и что нужно добавить перед реальным запуском.

## 1. Какие данные есть на платформе

### Публичные данные

Эти данные можно показывать другим пользователям:

- имя или ник исполнителя;
- рейтинг;
- описание услуги;
- портфолио;
- цена услуги;
- публичные карточки заказов без личной информации.

### Личные данные

Эти данные нельзя показывать всем подряд:

- email;
- телефон;
- история заказов пользователя;
- личная переписка;
- файлы технического задания;
- файлы результата;
- история операций;
- обращения в поддержку.

### Секретные данные

Эти данные не должны попадать в frontend, GitHub и логи:

- пароль;
- токены авторизации;
- ключи платежного провайдера;
- connection string;
- cookie сессии;
- admin-token;
- webhook-secret.

## 2. Главное правило проекта

**Храним только то, что реально нужно для работы биржи.**

Для MVP достаточно:

```text
UserId
UserName
Email
Role
OrderId
OfferId
PaymentStatus
ProviderPaymentId
CreatedAt
```

Не нужно хранить:

```text
номер карты
CVC
паспорт
адрес проживания
личные документы
лишние банковские данные
```

## 3. Безопасная схема платежей

Правильная схема:

```text
Пользователь оплачивает заказ через платежного провайдера
↓
Провайдер возвращает ProviderPaymentId
↓
ReviMarket сохраняет только статус платежа
↓
Данные карты остаются у платежного провайдера
```

В нашей базе:

```text
Payments
- Id
- OrderId
- UserId
- Amount
- Commission
- DesignerAmount
- Status
- ProviderPaymentId
- CreatedAt
```

## 4. Доступ к заказам

Права доступа:

| Роль | Что можно |
|---|---|
| Customer | создавать заказ, видеть свои заказы, выбирать исполнителя |
| Designer | видеть открытые заказы, оставлять отклики, видеть свои работы |
| Admin | модерировать заказы, пользователей и споры |

Обязательное правило:

```text
Пользователь не может открыть чужой приватный заказ по прямой ссылке.
```

## 5. Сообщения и чат

Сообщения должны быть видны только участникам заказа и администратору при споре.

Нельзя:

- показывать чат в публичном списке;
- отдавать все сообщения всем пользователям;
- писать полный текст приватных сообщений в обычные логи.

Можно логировать только технические события:

```text
MessageCreated: OrderId=15, SenderId=4, CreatedAt=...
```

## 6. Файлы пользователя

Перед реальным запуском добавить:

- проверку расширения;
- проверку MIME-типа;
- ограничение размера;
- случайное имя файла;
- запрет выполнения загруженных файлов;
- отдельную папку для uploads;
- доступ к файлам только участникам заказа.

Запрещённые расширения:

```text
.exe
.bat
.cmd
.js
.php
.html
.cshtml
```

## 7. Frontend beta и localStorage

В текущей beta данные хранятся в браузере через `localStorage`.

Правила:

- не хранить настоящие пароли;
- не хранить реальные платежные данные;
- перед сохранением спрашивать согласие пользователя;
- давать кнопку очистки demo-данных;
- очищать пользовательский ввод перед выводом на страницу.

## 8. Backend roadmap

Для настоящей версии добавить:

```text
ASP.NET Core MVC / Web API
ASP.NET Core Identity
SQL Server
Entity Framework Core
Roles: Customer / Designer / Admin
Payments via provider API
Webhook signature validation
File upload validation
Audit logs
Admin panel
Privacy policy
Terms of service
```

## 9. Минимальные страницы для доверия

На сайте должны быть:

- Политика конфиденциальности;
- Пользовательское соглашение;
- Правила безопасной сделки;
- Правила возврата;
- Контакты поддержки;
- Удаление demo-данных / аккаунта.

## 10. Перед публикацией настоящей версии

Проверить:

- [ ] секреты не лежат в GitHub;
- [ ] checkout/payment работает только через HTTPS;
- [ ] админка закрыта ролью `Admin`;
- [ ] пользователь видит только свои приватные данные;
- [ ] файлы не исполняются на сервере;
- [ ] платежные webhooks проверяются по подписи;
- [ ] есть backups базы данных;
- [ ] есть логирование ошибок;
- [ ] есть политика обработки персональных данных;
- [ ] настоящие платежи тестируются сначала в sandbox/test mode.
