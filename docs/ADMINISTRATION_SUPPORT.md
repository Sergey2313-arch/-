# Администрирование и поддержка — ReviMarket

Этот модуль нужен, чтобы биржу могли модерировать не только владельцы, но и отдельные сотрудники поддержки.

## Зачем нужна система администрации

На бирже будут заказчики, исполнители, заказы, файлы, чаты, платежи и споры. Без админ-панели проект быстро станет неуправляемым.

Админ-система должна помогать:

- проверять жалобы;
- модерировать заказы и товары;
- блокировать мошенников;
- отвечать пользователям в поддержке;
- решать споры между заказчиком и исполнителем;
- смотреть историю действий;
- контролировать платежи и возвраты;
- следить за безопасностью.

## Роли администрации

| Роль | Возможности |
|---|---|
| Owner | полный доступ ко всему проекту |
| Admin | управление пользователями, заказами, спорами, настройками |
| Moderator | проверка заказов, товаров, жалоб и блокировка контента |
| SupportAgent | ответы в поддержку, помощь пользователям, создание тикетов |
| FinanceManager | просмотр платежей, комиссий, возвратов и выплат |
| ReadOnly | только просмотр статистики и логов без изменений |

## Главные разделы админ-панели

### 1. Dashboard

Показывает:

- количество пользователей;
- активные заказы;
- открытые тикеты;
- жалобы;
- споры;
- платежи;
- комиссию платформы;
- подозрительную активность.

### 2. Модерация заказов

Модератор видит новые и пожалованные заказы.

Действия:

- одобрить;
- скрыть;
- отправить на правку;
- удалить;
- заблокировать автора;
- открыть спор.

### 3. Модерация исполнителей

Проверка профилей дизайнеров и фрилансеров.

Действия:

- подтвердить профиль;
- скрыть портфолио;
- отправить профиль на правку;
- выдать предупреждение;
- временно заблокировать;
- полностью заблокировать.

### 4. Тикеты поддержки

Пользователь пишет в поддержку, создаётся тикет.

Статусы тикета:

```text
New
InProgress
WaitingUser
Resolved
Closed
```

Приоритеты:

```text
Low
Medium
High
Critical
```

### 5. Споры по сделкам

Когда заказчик и исполнитель не договорились, открывается спор.

Админ смотрит:

- заказ;
- переписку;
- файлы;
- сроки;
- платеж;
- действия сторон.

Решения:

- деньги заказчику;
- деньги исполнителю;
- частичный возврат;
- продлить срок;
- отправить на доработку;
- заблокировать нарушителя.

### 6. Платежи и комиссии

FinanceManager видит:

- оплаченные заказы;
- замороженные средства;
- комиссии платформы;
- выплаты исполнителям;
- возвраты;
- ошибки платежей.

Важно: админка не должна показывать номер карты, CVC и лишние банковские данные.

### 7. Audit log

Лог действий нужен, чтобы понимать, кто и что сделал.

Примеры событий:

```text
AdminBlockedUser
ModeratorApprovedOrder
SupportAgentResolvedTicket
FinanceManagerCreatedRefund
UserOpenedDispute
```

В логах хранить:

```text
Id
ActorUserId
ActorRole
Action
TargetType
TargetId
CreatedAt
IpAddress
Comment
```

Нельзя писать в логи:

- пароль;
- токены;
- CVC;
- полный номер карты;
- приватные сообщения целиком;
- секретные ключи.

## Пример модели тикета

```csharp
public class SupportTicket
{
    public int Id { get; set; }
    public string UserId { get; set; } = string.Empty;
    public string? AssignedAgentId { get; set; }
    public string Subject { get; set; } = string.Empty;
    public string Status { get; set; } = "New";
    public string Priority { get; set; } = "Medium";
    public DateTime CreatedAt { get; set; } = DateTime.UtcNow;
    public DateTime? ClosedAt { get; set; }
}
```

## Пример проверки роли

```csharp
[Authorize(Roles = "Owner,Admin,Moderator")]

    return View();
```

```csharp

{
    return View();
}
```
